# F-3.5: Filtered Portmapper Bypass via Local Tunnel

## Classification

- **Severity**: Medium
- **CVSS Vector**: Network / Low Complexity / No Auth Required
- **Affected Versions**: Any NFS server with port 111 filtered but NFS ports open
- **Prerequisite**: NFS ports (2049, mountd) reachable; port 111 filtered by firewall

## Summary

When a firewall filters port 111 (portmapper) but leaves NFS-related ports (2049, mountd) open, standard tools like `showmount` and `rpcinfo` fail because they first contact portmapper to discover service ports. By simulating a portmapper service locally and creating a tunnel, an attacker can redirect portmapper queries to the target's NFS ports directly, bypassing the filtered state entirely.

## Technical Detail

### The Problem

Standard NFS tools follow this flow:
1. Contact portmapper (111) → "where is mountd?"
2. Portmapper replies: "mountd is on port 34567"
3. Contact mountd (34567) → "list exports"
4. Contact nfsd (2049) → mount and access files

When port 111 is filtered, step 1 fails and the entire chain breaks — even though NFS itself is reachable.

### The Bypass

The attacker runs a fake portmapper locally that returns hardcoded port mappings:

```bash
# 1. Discover open NFS-related ports via direct scanning
nmap -p 2049,111,20048 -sV target
# Result: 2049/open, 111/filtered, 20048/open (mountd)

# 2. Create SSH tunnel or socat relay for NFS ports
socat TCP-LISTEN:2049,fork TCP:target:2049 &
socat TCP-LISTEN:20048,fork TCP:target:20048 &

# 3. Run a local fake portmapper that maps:
#    - Program 100003 (NFS) → localhost:2049
#    - Program 100005 (MOUNT) → localhost:20048
# Various tools exist for this, or use rpcbind with manual registration

# 4. Point standard tools at localhost
showmount -e localhost
mount -t nfs localhost:/export /mnt
```

### Alternative: Direct Mount Without Portmapper

Some NFS clients support specifying the mountd port directly:

```bash
# Skip portmapper entirely by specifying mountport
mount -t nfs -o mountport=20048,port=2049 target:/export /mnt

# Or with nfsshell / nfswolf:
nfswolf shell target:/export --mount-port 20048 --nfs-port 2049
```

### Why Firewalling Port 111 Is Insufficient

- NFS ports can often be guessed or scanned (2049 is standard, mountd is often 20048 or in the 32768-60999 ephemeral range)
- The portmapper is a convenience for the client, not a security gate
- Blocking it gives a false sense of security while the actual services remain exposed

## Impact

- Access NFS exports that were "protected" by filtering portmapper
- Full exploitation chain still works: mount, UID spoof, file access
- False sense of security for administrators who only firewall port 111

## Detection (nfswolf)

The scanner should:
1. When port 111 is filtered, still scan common NFS ports (2049, 20048, ephemeral range)
2. Attempt direct mount without portmapper using `--mount-port` / `--nfs-port` overrides
3. Report when NFS is reachable despite filtered portmapper (misconfigured firewall)

## Remediation

1. **Firewall ALL RPC ports**, not just portmapper — block 2049, mountd, NLM, NSM
2. **Use NFSv4 only** — v4 uses a single port (2049), no portmapper needed, but still must be firewalled
3. **Network segmentation** — NFS should only be reachable from trusted VLANs
4. **Don't rely on port 111 filtering** as a security measure for NFS
